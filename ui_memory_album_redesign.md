# 回忆图鉴重设计 — 规划

## 问题陈述

「回忆图鉴」当前实现是一份 **AI 自动生成的回忆录 + TTS 朗读**：
- 开篇文字段 → 老人简介卡 → 多个章节（章节标题 + 章节简介 + 多个 item，每个 item 又是标题+正文）→ 时间轴 → 给家人的问题 → 注释 → 结束语 → 朗读控制条
- 照片仅作为某些 item 的配图穿插在大段叙事文本之间

**为什么觉得奇怪**（设计层面）：
1. **名实不符**：「图鉴」字面是 visual gallery，实际是 text-heavy biography
2. **认知负载方向错了**：阿尔兹海默症用户读长段落困难，但**识别照片中熟悉的脸**反而能触发记忆 —— 这是医学共识。文字主导的方案把信息架构倒置了。
3. **照片被埋**：每张照片夹在两段 AI 生成的散文中间，难以扫视/快速浏览
4. **AI 越界了**：AI 应该**帮助回忆**（提供线索），不是**替代回忆**（自己讲一遍）

**当前实现里好的部分**（不应该删）：
- TTS 朗读 + 句子级高亮（技术成熟，有价值）
- 数据模型完整（chapters / items / segments / 时间轴 / 给家人的问题）
- "给家人的问题" 这个功能对照护者真的有用
- 仓库层 `MemoryAlbumRepository` 把所有数据已经组织好了

## 设计目标（按优先级）

1. **照片优先**：默认看到的是照片，不是文字
2. **保留 AI 能力**：朗读 / 章节 / 问家人这些功能不删，但**降为"模式"**，不是默认 chrome
3. **老人独立可用**：触屏可以滑、点击区域大、不需要读小字就能用
4. **照护者也能用**：可以快速浏览老人的一生 / 也可以打开"听故事"模式陪老人一起听
5. **保持现有调性**：原版青绿 + 奶油配色不变（你已经定调）

## 三个候选方向

### A. 照片墙（Pinterest 风网格）⭐ 推荐

```
┌─────────────────────────────────┐
│  回忆图鉴          [听] [刷新]  │  ← AppBar 已有，这里加两个角按钮
├─────────────────────────────────┤
│ [全部] [家庭] [经历] [日常]    │  ← 分类 chips
├─────────────────────────────────┤
│  ┌─────────┐   ┌─────────┐    │
│  │ 老照片  │   │  照片   │    │
│  │         │   │         │    │
│  │  1986   │   │  1992   │    │
│  └─────────┘   └─────────┘    │
│   春天里的车    女儿满月        │  ← 1 行 caption
│                                 │
│  ┌─────────┐   ┌─────────┐    │
│  │   ...   │   │   ...   │    │
└─────────────────────────────────┘
```

**特点**：
- 2 列瀑布流（masonry），每张照片大小自适应
- 每个卡片：照片 + 1 行 caption + 小日期 pill
- 顶部分类 chips（基于现有 `ProfilePhotoCategory`）
- 右上角「听」按钮 → 打开朗读模式（沿用现 narration 实现）
- 空状态：温柔提示 + "去预录入加照片"按钮
- **点击照片** → 弹出底部 sheet，放大照片 + AI 整理的 caption + "听这一段"按钮

**优点**：
- 最像市面成熟相册（Photos.app / 朋友圈相册 / Pinterest）
- 一屏能看到多张照片，触发回忆效率高
- 没有照片的 memory item（纯文字事件）也可以做成"文字卡"混在网格里，不会浪费数据
- 实现风险最低

**缺点**：
- "讲故事的连贯感"被打散（不过通过「听」按钮可以恢复）

---

### B. 翻页相册（PageView 横滑）

```
┌─────────────────────────────────┐
│              < 3 / 24 >          │  ← 页码
├─────────────────────────────────┤
│                                 │
│  ┌───────────────────────┐     │
│  │                       │     │
│  │     大 照 片          │     │
│  │                       │     │
│  │                       │     │
│  └───────────────────────┘     │
│                                 │
│  春天里的自行车                 │  ← 标题
│  1986 · 老家小院                │  ← 时间地点
│                                 │
│  那年外公给我买了第一辆...      │  ← 一两句话
│                                 │
│  ● ● ○ ○ ○ ○                  │  ← 进度点
└─────────────────────────────────┘
```

**特点**：
- 一次只看一张照片（沉浸感）
- 横向滑动翻页（像真的相册）
- 文字克制（标题 + 时空 + 一两句）

**优点**：
- 最有"翻看相册"的物理感
- 适合慢节奏陪伴（爷爷奶奶坐下来一张张看）
- 文字量天然受限于单页空间，不会膨胀

**缺点**：
- 浏览效率低（看 20 张要滑 20 次）
- 不容易跳转到某一张特定照片
- 实现稍复杂（PageView + 滑动动画）

### C. 时光卷轴（垂直时间线）

```
┌─────────────────────────────────┐
│  回忆图鉴            [听]       │
├─────────────────────────────────┤
│  1946                           │
│  ├─ ┌──────────┐               │
│  │  │  童年    │ 出生在江南     │
│  │  └──────────┘                │
│  │                              │
│  1965                           │
│  ├─ ┌──────────┐               │
│  │  │  结婚    │ 在老家结婚     │
│  │  └──────────┘                │
│  │                              │
│  1986                           │
│  ├─ ┌──────────┐               │
│  │  │  自行车  │ 春天里的...    │
│  │  └──────────┘                │
└─────────────────────────────────┘
```

**特点**：
- 左侧年代刻度，右侧照片 + 简短说明
- 垂直滚动一条线讲完一生

**优点**：
- "看一遍 ta 的一生" 最直观
- 适合照护者初次浏览了解老人

**缺点**：
- 强依赖日期数据完整性（很多老照片没准确年份）
- 老人自己用：从 1946 滚到 2025 路径长
- 时间轴线条 + 年代刻度会让画面看起来"严肃"，不够温暖

## 推荐路径

**Phase F-1：先做 A（照片墙）作为新默认视图**
- 砍掉默认显示的所有 _Album*Panel / _Narration*Panel
- 加一个右上角「听故事」按钮 → 打开新的 `_NarrationModeView`，把现在的所有 panel 移过去
- 默认视图 = 分类 chips + 瀑布流照片网格
- 点击照片 → 底部 sheet 显示详情（大图 + caption + "听这一段"）

**Phase F-2（可选）：做 B（翻页相册）作为另一种模式**
- 加一个「相册模式」切换按钮
- 在网格 / 翻页之间切换
- 看老人 / 家属偏好哪种

**Phase F-3（可选）：C 留到最后**
- 如果日期数据足够好且家属反馈想看，再做
- 否则就不做

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| 用户找不到原来的"AI 讲故事"功能 | 右上角「听」按钮 + 首次进入弹一次气泡引导"想听故事？点这里" |
| 照片很少（< 6 张）网格看起来空 | 把 memory event 也变成"文字卡"混排，没照片的卡显示米色底 + 标题文字 |
| 老人误触底部 sheet 摸不到关闭 | sheet 显示半屏，留下顶部 40% 显示原页面，下拉即关 |
| narration 朗读功能体验断裂 | narration mode 进入后仍是完整的现有体验，零功能损失 |
| 现有 narration 测试用例破坏 | 不动 `_AlbumChapterPanel` 等类的内部实现，只是移动它们被哪个视图渲染 |

## 我推荐的具体方案（用 frontend-design skill 视角）

**承诺一个方向**：相册体验 = 照片墙（A）。不试图同时兼顾"AI 回忆录"和"照片浏览"两个矛盾的角色。

**dominant 元素**：照片本身（画面 70% 以上）。文字 caption 是辅助（不超过 1 行）。

**sharp accent**：分类 chips 用主色青绿，照片日期 pill 用强调橙 `#F4A261`（之前你的强调橙用得太少，正好给它一个明确角色 = "时间戳"）。

**记忆点**：每张照片有"轻描边 + 暖白底 + 1px hairline"形成"小相片"质感，配上 1 行手写感的 caption，整页像一面贴着照片的墙。

**克制处**：
- 不做 polaroid 翻转动画
- 不做照片小尺寸的预加载模糊（直接显示，性能优先）
- 不做"点赞 / 收藏"这种社交元素（与场景不符）

## 等你确认的事项

1. **方向**：A（照片墙）/ B（翻页）/ C（时间轴）选哪个做 Phase F-1？我推荐 **A**。
2. **朗读模式入口**：右上角小图标 OK 吗？还是想要更醒目的"听故事" 大按钮（比如顶部一个 banner）？
3. **没有照片的 memory event**（纯文字"重要经历"）：
   - 选项 a：作为文字卡混在网格里
   - 选项 b：完全隐藏，回忆图鉴只展示有照片的
4. **现在显示的"开篇/结束语/给家人的问题/注释"** 这些 AI 生成的文字段：
   - 选项 a：全部转移到朗读模式
   - 选项 b：部分删除（比如开篇结束语这种装饰性的）
   - 选项 c：保留全部，让"朗读模式"成为很完整的功能页

确认后我开始 Phase F-1 实施。
